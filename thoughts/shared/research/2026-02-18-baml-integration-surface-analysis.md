# Research: BAML Integration Surface Analysis

**Date**: 2026-02-18T16:55:43Z
**Researcher**: Claude
**Git Commit**: f057ede731d6e7eded7fb2ca4f6b4bcde441f7dc
**Branch**: main
**Repository**: autopoiesis

## Research Question

Document the integration layer, provider architecture, capability system, extension compiler, and prompt registry to understand the integration surface for adding BAML (BoundaryML) as a typed prompt-to-structured-output layer.

## Summary

The Autopoiesis integration layer is a mature, multi-layered system with five distinct integration patterns: (1) direct HTTP API calls via `llm-complete` CLOS dispatch, (2) CLI provider subprocesses via a macro-generated provider hierarchy, (3) an inference provider that wraps the agentic loop as a provider, (4) MCP server stdio JSON-RPC connections, and (5) a self-extension compiler for agent-written code. The system has a unified capability registry that bridges all of these to a single tool-call dispatch mechanism. A versioned prompt registry with templating, forking, and substrate persistence already exists. There is no FFI layer — all external integration uses subprocesses or HTTP.

## Detailed Findings

### 1. Provider Architecture

The provider system has three tiers:

**Base Provider** (`platform/src/integration/provider.lisp:13-68`): Abstract CLOS class with generic functions `provider-invoke`, `provider-build-command`, `provider-parse-output`, `provider-format-tools`. Default `provider-invoke` calls `run-provider-subprocess` which uses `sb-ext:run-program` with two reader threads for stdout/stderr.

**CLI Providers** (generated by `define-cli-provider` macro at `platform/src/integration/provider-macro.lisp:103-236`): Four concrete providers — Claude Code, Codex, Cursor, OpenCode — each generated from a declarative specification. Two parser strategies:
- `:json-object` — parses entire stdout as single JSON (Claude Code, Cursor)
- `:jsonl-events` — line-by-line JSON event stream with type dispatch (Codex, OpenCode)

**Inference Provider** (`platform/src/integration/provider-inference.lisp:19-43`): Subclass that makes direct HTTP API calls instead of spawning subprocesses. Wraps the `agentic-loop` directly. Supports Anthropic, OpenAI, and Ollama endpoints via `api-format` dispatch (`:anthropic` or `:openai`).

**Provider Registry** (`provider.lisp:204-231`): Global `*provider-registry*` hash table with `register-provider`, `find-provider`, `list-providers`. Providers are registered by name string.

### 2. LLM Client Protocol

**File**: `platform/src/integration/llm-client.lisp`

Two CLOS generics define the protocol:
- `llm-complete (client messages &key tools system)` — the single dispatch point
- `llm-auth-headers (client)` — provider-specific auth headers

**Claude Client** (`platform/src/integration/claude-bridge.lisp:11-39`): Posts to `/v1/messages` with `x-api-key` auth. Default model: `claude-sonnet-4-20250514`.

**OpenAI Client** (`platform/src/integration/openai-bridge.lisp:14-39`): Posts to `/v1/chat/completions` with `Bearer` auth. **Normalizes responses back to Claude format** via `openai-response-to-claude-format`, so the agentic loop operates on a single response shape.

**Message Format Conversion** (`openai-bridge.lisp:55-117`): `claude-messages-to-openai` and `claude-tools-to-openai` handle the bidirectional format mapping. Tool call format differences (Claude `tool_use` content blocks vs OpenAI `tool_calls` array) are transparently converted.

### 3. Agentic Loop

**File**: `platform/src/integration/claude-bridge.lisp:163-214`

`agentic-loop` is the core multi-turn tool-use cycle:

```
LOOP:
  1. llm-complete(client, messages, :system, :tools)
  2. Append assistant message to conversation
  3. If stop-reason = "tool_use" AND turn-count < max-turns:
       execute-all-tool-calls → format-tool-results → append to messages → GOTO 1
  4. Else: return (values response messages turn-count)
```

Test override: `*claude-complete-function*` dynamic variable (`claude-bridge.lisp:158`) allows complete replacement of the LLM call for testing.

The `agentic-agent` class (`platform/src/integration/agentic-agent.lisp:53`) wires the cognitive loop phases to the agentic loop, recording turns to substrate and maintaining conversation history.

### 4. Capability System

**Base Class** (`platform/src/agent/capability.lisp:11-30`): Slots: `name` (symbol), `function` (callable), `parameters` (list of specs), `permissions` (keyword list), `description` (string).

**Global Registry** (`capability.lisp:45`): `*capability-registry*` hash table with `register-capability`, `find-capability`, `invoke-capability`.

**`defcapability` Macro** (`capability.lisp:138-171`): Primary extension point. Expands to `register-capability` at load time. Parses lambda lists into parameter specs with type, required, and default metadata.

**Agent-Defined Capabilities** (`platform/src/agent/agent-capability.lisp:12-44`): Extends `capability` with `source-agent`, `source-code`, `extension-id`, `promotion-status` (`:draft` -> `:testing` -> `:promoted`/`:rejected`). The lifecycle: `agent-define-capability` -> `test-agent-capability` -> `promote-capability`.

**Built-in Capabilities**: 4 cognitive capabilities (introspect, spawn, communicate, receive) + 22+ tool capabilities (file ops, web, shell, git, self-extension, orchestration).

### 5. Capability-to-Tool Mapping

**File**: `platform/src/integration/tool-mapping.lisp`

**Outbound (capability -> Claude tool spec)**: `capability-to-claude-tool` (line 143) produces `{"name": "snake_case", "description": "...", "input_schema": <JSON Schema>}`. Type mapping: Lisp `string`->`"string"`, `integer`->`"integer"`, `list`->`"array"`, etc.

**Inbound (tool call -> capability invocation)**: `execute-tool-call` (line 209) converts snake_case name to keyword, looks up capability, converts JSON alist input to keyword args, and `apply`s the function.

**Name Convention Bridge**: `lisp-name-to-tool-name` (kebab->snake) and `tool-name-to-lisp-name` (snake->keyword) at lines 12-24.

### 6. Extension Compiler

**File**: `platform/src/core/extension-compiler.lisp`

**Sandbox Validation** (lines 64-197): Whitelist-based code walker checks operator-position symbols against:
- `*allowed-packages*` — CL, KEYWORD, AUTOPOIESIS.CORE, AUTOPOIESIS.AGENT, ALEXANDRIA
- `*forbidden-symbols*` — eval, compile, load, open, delete-file, run-program, setf, defun, etc.
- `*sandbox-allowed-symbols*` — safe operations (car, cdr, cons, mapcar, arithmetic, etc.)
- Three sandbox levels: `:strict` (full checks), `:moderate` (relaxed package checks), `:trusted` (no checks)

**Compilation** (`compile-extension`, line 389): Wraps source in `(lambda () ,source)`, calls `(compile nil ...)` for native compilation. Extensions stored in `*extension-registry*` by UUID.

**Auto-disable**: Extensions exceeding 3 runtime errors are automatically set to `:rejected` status.

### 7. Prompt Registry

**File**: `platform/src/integration/prompt-registry.lisp`

A first-class versioned prompt system with:
- **`prompt-template` class** (line 14): name, category, body, version, content-hash (SHA256), parent hash (for fork lineage), author, variables, includes
- **Registry** (`*prompt-registry*`, `*prompt-history*`): name->current and name->version-list hash tables
- **Templating**: `{{variable}}` substitution and `{{include:prompt-name}}` composition with cycle detection
- **Forking**: `fork-prompt` creates derived prompts linking to parent via content-hash
- **Substrate Persistence**: `persist-prompt` writes prompts as datoms; `load-prompts-from-substrate` restores them
- **`defprompt` Macro** (line 313): Load-time prompt definition and registration
- **5 Built-in Prompts**: `agent-guidelines`, `cognitive-base`, `self-extension`, `provider-bridge`, `orchestration`

### 8. MCP Client

**File**: `platform/src/integration/mcp-client.lisp`

Spawns MCP server processes via `sb-ext:run-program` with bidirectional stdio streams. Uses newline-delimited JSON-RPC 2.0 transport. Auto-discovers tools and resources on connect. **MCP tools are auto-bridged to capabilities** via `mcp-tool-to-capability` (line 398) and `register-mcp-tools-as-capabilities` (line 424).

### 9. External Integration Patterns

**No FFI/Rust**: Zero CFFI calls. The holodeck is pure CL using `cl-fast-ecs`. GLSL shaders are stored as strings with no GPU backend.

**Three subprocess patterns**:
1. Orchestration Claude worker: `sb-ext:run-program "/bin/sh" ["-c" cmd]` with `</dev/null`
2. Provider system: `sb-ext:run-program command args` with two reader threads
3. MCP: `sb-ext:run-program` with bidirectional `:stream` I/O

**JSON parsing**: `cl-json:decode-json-from-string` throughout. Note: `cl-json` converts `snake_case` to `:SNAKE--CASE` (double-hyphen), which is handled by the provider macro's dual-key lookup.

**No schema validation of external responses**: All parsing is `assoc`-based alist navigation; missing keys return `nil` silently.

### Name Convention Flow

```
Lisp definition:  (defcapability read-file ...)
                       |
                  symbol 'READ-FILE
                       |
              lisp-name-to-tool-name
                       |
              "read_file"  (sent to LLM)
                       |
              [LLM calls tool "read_file"]
                       |
              tool-name-to-lisp-name
                       |
              :READ-FILE  (keyword)
                       |
              find-capability (with string= fallback)
                       |
              (apply capability-function args)
```

### Extension Compiler Pipeline

```
Agent S-expression → validate-extension-source
                        ├── check forbidden patterns
                        ├── check operator packages
                        ├── check operator whitelist
                        └── recursive form walk
                              |
                     (compile nil (lambda () source))
                              |
                     extension instance (:validated)
                              |
                     *extension-registry* (keyed by UUID)
                              |
                     invoke-extension (with error counting)
                              |
                     auto-reject at > 3 errors
```

### Prompt Template Pipeline

```
defprompt / make-prompt-template
        |
   register-prompt (auto-version, link parent hash)
        |
   *prompt-registry* + *prompt-history*
        |
   render-prompt
        ├── resolve-includes ({{include:name}} with cycle detection)
        └── substitute-variables ({{variable}} from bindings alist)
        |
   persist-prompt → substrate datoms (EAV triples)
        |
   fork-prompt → new prompt linked to parent via content-hash
```

## Open Questions

1. **BAML compilation target**: BAML compiles to host-language clients. The existing integration patterns are all subprocess-based or HTTP-based. What would the BAML CL integration look like — subprocess calling `baml-cli`, a CFFI binding to the Rust runtime, or a pure-CL reimplementation of the BAML parser/SAP engine?

2. **Schema validation gap**: The current system does no validation of LLM responses against expected schemas. All parsing is `assoc`-based with silent `nil` on missing keys. BAML's Schema-Aligned Parsing (SAP) would address this, but the integration path depends on whether SAP runs in-process or as an external tool.

3. **Prompt registry convergence**: The `prompt-registry` already supports versioned, composable, forkable prompts with substrate persistence. How would `.baml` file schemas relate to `prompt-template` instances? Are they complementary (BAML defines the output schema, prompt-template defines the prompt text) or overlapping?

4. **Capability type metadata**: The current `capability-params-to-json-schema` maps Lisp types to JSON Schema types, but the mapping is coarse (everything defaults to `"string"`). BAML's rich type system (classes, enums, unions, optionals) would need a richer type representation in the capability parameter specs.

5. **Streaming integration**: The agentic loop is synchronous (blocking `llm-complete` calls). BAML supports type-safe streaming with partial results. The current architecture would need extension for streaming structured output.
